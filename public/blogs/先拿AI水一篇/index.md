很多人都遇到过这种离谱场景：你明明复制好了代码，Ctrl+V 一按，网站就是不让粘贴。
更离谱的是，你用自动化工具模拟键盘输入，结果中文输入法一介入，整段代码直接乱码。

所以我做了一个小工具：KeyCast（键影）。目标很简单：
把任意文本“像真实键盘敲击一样”稳定输入到目标窗口，同时尽可能不被输入法和网页编辑器干扰。

为什么这个问题很难
表面看只是“自动打字”，实际有三个坑：

第一，输入法干扰。
很多库是按虚拟键码发按键，系统会把它交给 IME（输入法）处理，最后出来的是拼音候选，不是你原文。

第二，代码不是普通文本。
普通文本只要字对就行；代码还要求换行、缩进、Tab 层级都对。一旦编辑器自动缩进和模拟输入打架，结构立刻乱掉。

第三，网页编辑器行为不一致。
有的编辑器回车会自动补齐缩进，有的会补全括号，有的最后一行处理还不一样。
同一套策略在 A 站好用，在 B 站可能翻车。

KeyCast 的核心实现思路
我最后采用的是：底层 SendInput + GUI 线程隔离 + 可切换的代码模式策略。

1）底层输入：绕开输入法
通过 ctypes 直接调用 Windows user32.SendInput，并在普通字符输入时使用 KEYEVENTF_UNICODE。
这一步的意义是：字符直接以 Unicode 进入输入管线，而不是走传统键盘扫描码路径，最大限度降低 IME 影响。

2）代码场景：把“换行和缩进”当一等公民
工具对输入内容逐字符处理，并区分：

换行
Tab
普通字符
同时提供代码模式来应对不同编辑器的自动缩进机制，避免“越敲越偏”或最后一行错位。

3）桌面体验：不是脚本，是可用工具
我用 CustomTkinter 做了完整 GUI，包括：

剪贴板导入
启动延迟（滑块 + 输入框联动）
输入速度控制
倒计时进度条
状态提示
Esc 紧急中断
并且输入任务放在守护线程中，主界面不冻结，这点对可用性非常关键。
