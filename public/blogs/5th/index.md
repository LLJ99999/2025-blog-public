最近，在闲鱼上买了个机械臂，买这台机械臂的时候，本来只是想把它跑起来，顺便做一些抓取和自动化的小实验。结果真正拿到手之后，我很快发现，硬件只是开始，真正麻烦的是控制链路。

这台机械臂买回来的时候，板子里刷的是卖家的上下位机方案。它能用，但核心问题是：闭源，而且完整软件要额外付费。对我来说，这件事有点别扭。机械臂我已经买了，但真正的控制能力却被锁在一套我看不到、也改不了的系统里。

所以后来我做了两件事。第一步，是把固件刷回开源版本。第二步，是自己从零做一套上位机。

我这篇文章不打算讲“我做了一个很厉害的软件”，而是想比较真实地复盘：我是怎么从一个几乎只能基础控制的状态，慢慢把它做成现在这个有手动控制、示教编排、气泵联动、3D 预览、视觉接入，甚至已经开始尝试 AI Agent 的上位机系统。

## 一开始，我并不是直接写桌面上位机

最早我其实走过一条弯路。

我先让 AI 帮我快速写了一个网页版的控制 MVP。当时想法很自然：网页开发快，界面出得也快，先验证控制逻辑，再慢慢做完整系统。但这个方案最后并没有真正跑通。原因倒也不复杂，机械臂这种东西和普通网页交互不一样，它不是“点一下按钮返回个结果”这么简单，而是牵涉到串口通信、动作时序、设备状态、连续执行这些问题。网页能很快做出界面，但不太适合我当时那个阶段去做底层调试。![](https://i.imgs.ovh/2026/04/29/jIOcre.png)

所以后来我换了思路，不再先做“看起来像上位机”的东西，而是先去摸清它到底是怎么通信、怎么执行动作的。

## 真正把项目拉起来的，是我先把通信链路摸明白了

那段时间我做的事情，说白了就是两个字：测试。

我一开始是直接用串口工具去和控制板通信，一条一条命令试。机械臂这种设备，很多事情你看文档不一定看得出来，但你真的去发命令、看返回、改参数，很快就会知道它的工作方式是什么。

也是在这个阶段，我慢慢摸清了它的动作控制模式。它并不是我一开始以为的那种“给一个目标坐标，它自己就完整做完抓取”的黑盒流程，而是更接近一套拆开的动作链：移动、吸气、释放这些动作，本质上是分开的，需要由上位机自己去组织和编排。

在我现在的代码里，这件事最后被抽象成了两类最基本的动作：

- `move`：移动到某个 `X/Y/Z`
- `pump`：控制气泵开或者关

这也是为什么我后面的示教器、预设系统、AI 规划，最后都能统一到一套动作结构上。因为我不是先做了一个复杂界面，而是先把“机械臂到底会执行什么”这件事摸透了。

## 我的第一个可用版本，本质上是“手动控制 + 串口发指令”

当我明确了通信方式之后，第一个真正能用的版本其实并不复杂。它的核心就是一件事：在上位机里维护当前坐标状态，然后把坐标转换成串口命令发出去。

我现在的桌面端是用 `PySide6` 写的，主窗口类是 `WeaverHost`。在这个类里，我维护了一组最基本的位置状态：

```python
self.pos = {'X': 0, 'Y': 120, 'Z': 120}
```

无论我是按方向按钮、拖虚拟摇杆、点击归零，最终都会回到同一个出口函数：

```python
def update_hardware(self, force_send=False):
    self.clamp_position()
    px, py, pz = int(self.pos['X']), int(self.pos['Y']), int(self.pos['Z'])
    self.gl_widget.update_pose(px, py, pz)
    cmd = f"G1 X{px} Y{py} Z{pz} F{self.move_feedrate}"
    if force_send or cmd != self.last_sent_cmd:
        self.send_cmd(cmd)
```

这段逻辑其实很能代表我整个项目的思路。不是每个按钮各发各的命令，而是先统一更新状态，再由一个总出口负责：

1. 处理坐标限制  
2. 更新 3D 预览  
3. 拼接 G-code  
4. 发串口

这样后面功能一多，系统也不容易乱。

## 后来我发现，真正难的不是“控制”，而是“让多步动作可靠执行”

如果只是手动按按钮，其实上位机并不难写。真正麻烦的是，当我开始做示教器、动作流水线和气泵联动的时候，系统就从“发一条命令”变成了“管理一串命令”。

这个时候最容易掉进去的坑就是：上位机以为发完了，下位机其实还没做完。

我最早也用过最直接的方法，就是每一步动作发完之后 `sleep` 一下，假设过一两秒机械臂应该走到了，再执行下一步。这个做法短时间内看起来是能跑的，但只要路径一长、速度一改、动作一多，它就开始失真。终端看起来在执行，机械臂实际上却没有按这个节奏完成。

所以后来我把执行链路彻底重做了。

我在下位机固件里加了一个很关键的完成反馈：`DONE`。机械臂每做完一个动作，就通过串口回传一条 `DONE`。上位机这边的 `SerialWorker` 专门监听这个消息：

```python
if data.upper() == "DONE":
    self.done_signal.emit()
```

然后动作执行线程 `SequenceWorker` 不再靠 `sleep`，而是老老实实等这个信号：

```python
if not self._done_event.wait(self.wait_timeout):
    self.log_signal.emit("步骤等待硬件完成超时", "error")
```

这一步改完之后，整个系统一下子稳定了很多。因为动作链路从“猜它做完了”变成了“它明确告诉我做完了”。

这也是我现在回头看，觉得整个项目里最重要的一次升级。它不是多加了一个功能，而是让整个上位机从“能发命令”，变成了“能调度任务”。

## 示教器是我最喜欢的一部分，因为它让机械臂第一次变得“可复现”

手动控制解决的是“我现在想让它怎么动”，示教器解决的是“我下次还想让它这样动”。

在我的实现里，示教区记录的不是原始串口文本，而是结构化动作列表。比如：

```python
{"type": "move", "x": 0, "y": 120, "z": 120}
{"type": "pump", "state": True}
```

所以我在界面里做了三个最基础的录制入口：

- 记录当前位置
- 记录吸气
- 记录释放

用户每点一次，就往 `current_sequence` 里追加一个动作对象。然后再由统一的 `SequenceWorker` 负责执行这一整条流水线。

这件事有两个明显好处。第一，逻辑清楚。第二，后面做预设和 AI 接入都很自然。因为动作本身已经不是“人类自己能看懂的一串命令”，而是机器也能直接处理的数据结构。

后来我还加了预设保存，直接把动作序列存成 JSON 文件 `weaver_presets.json`。这部分没有什么很复杂的算法，但它非常实用，而且特别适合面试时讲，因为它能体现出一个很重要的工程意识：不是只做一个“能点”的工具，而是让用户行为有沉淀。

## 气泵控制是个小模块，但它让我第一次有了“任务链”的感觉

机械臂只会移动，其实还不够。真正让我觉得“这玩意开始像个系统了”的，是我把气泵控制接进来之后。

下位机这边我给它加了两个命令：

- `M88`：开启气泵
- `M89`：关闭气泵

上位机里对应也非常直接：

```python
def execute_pump_action(self, state):
    self.send_cmd("M88" if state else "M89")
```

看起来就这么简单，但它的意义是动作模型完整了。因为到这里为止，我的上位机不再只是“坐标控制器”，而是能组织一个基本任务流了：

1. 抬到目标上方  
2. 下压  
3. 开启气泵  
4. 抬起  
5. 移动  
6. 关闭气泵

这也是后面 AI 规划为什么能接进来的基础。因为 AI 最终规划出来的不是抽象意图，而是这套已经在系统里存在的动作语言。

## 我后来还做了 3D 预览，而且不是只为了好看

这个项目里另一个我投入了不少时间的模块，是 3D 预览。

最开始的时候，我对它的预期其实很低，只想要一个姿态可视化，知道它当前往哪边伸就够了。但越往后做，我越发现，如果预览只是随便画几根线，它在调试时并没有太大价值。尤其是当我开始调结构参数、吸嘴显示、归零姿态时，一个不靠谱的预览甚至会反过来误导判断。

所以后来我单独写了 `Arm3DPreview`，用 `QOpenGLWidget` 去做渲染。这里面我不是简单画一条骨架，而是给机械臂拆成了底座、大臂、小臂、腕部和吸嘴，并且加了一套基于结构参数的姿态求解。

比如现在代码里有这些关键参数：

```python
self.base_h = 52.0
self.link_lower = 165.0
self.link_upper = 145.0
self.tool_off_r = 25.0
self.tool_off_z = 97.0
self.nozzle_draw_len = 24.0
```

后面我还专门把“参与 IK 的参数”和“纯视觉显示参数”拆开了。这个拆分非常重要，因为我中间踩过坑：为了让吸嘴看起来更短更直，我一开始直接改了姿态求解参数，结果整条机械臂的预览姿态都被我带歪了。后来我才意识到，运动学和渲染不能混为一谈。姿态怎么解是一回事，显示怎么画是另一回事。

这个过程其实很像一个小型工程训练：当系统开始变复杂之后，你必须不断地去划清边界，不然一个看似小的改动就会串到别的模块里。

## 视觉部分我已经接进来了，AI 也接了，但我对这块目前是克制的

在这个项目里，我确实已经把视觉和 AI 接进来了。

视觉部分目前比较朴素，用的是 OpenCV。相机线程 `CameraWorker` 做了背景建模、轮廓提取和目标中心识别。对我现在这个阶段来说，这已经够用了。因为我最重要的目标，不是做一个完美识别器，而是先把“图像输入”和“动作生成”这条链打通。

AI 部分我也接上了。现在它的实现方式是：

1. 获取当前相机画面  
2. 转成 base64  
3. 连同用户指令一起发给大模型  
4. 让模型返回结构化动作计划  
5. 动作进入 AI 预备区，再由我选择自动执行或手动逐步执行

在代码里，这块主要由 `AIAgentWorker` 负责。它做了两层兼容：一层是工具调用，一层是 JSON 文本回退解析。也就是说，就算模型没有严格按 tool call 输出，也还有容错余地。

不过说实话，这一部分我自己现在的评价是：方向是对的，但还没有到我满意的程度。尤其是 agent 的体验，现在还是有点卡，也还不够稳。它已经能工作，但还称不上成熟。所以我对这部分的态度是继续慢慢做，而不是急着把它包装成一个“AI 很强”的卖点。

我反而觉得，把这点诚实地写进博客里，对面试更有帮助。因为这说明你不是在堆 buzzword，而是真的知道一个模块现在做到什么程度、差距在哪、下一步怎么补。

## 回头看这个项目，我觉得最能说明问题的，不是功能数量，而是版本迭代方式

如果只是单看现在的界面，别人可能会觉得这就是一个“功能比较多的机械臂控制软件”。但对我自己来说，这个项目最有价值的地方，其实是它的迭代路径。

它不是我一口气设计好的完整系统，而是我一点点试出来的：

- 先用 AI 写网页版 MVP，发现不适合做底层控制  
- 再回到串口层，一条命令一条命令摸清协议  
- 再做最小可用的手动控制  
- 再做示教和动作流水线  
- 再加气泵控制  
- 再把串口执行从“时间驱动”改成“DONE 驱动”  
- 再做速度调节、动作预设、3D 预览  
- 最后再把视觉和 AI 一层层接进来

这条路线其实挺能代表我做项目的方式：不是先空想一个很大的架构图，而是先做一个能工作的版本，然后不断把最痛的那个问题解决掉。

我觉得这也是这篇文章最想传达给面试老师的一点：我不是在“搭一个能演示的作品”，而是在真的把一个系统做顺。

## 现在这套上位机对我来说，最重要的是它证明了我能把事情做到底

如果只是说“我会 Python”“我会 Qt”“我会调串口”，这些都太碎了。这个项目对我更大的意义，是它把这些能力串成了一条完整链路：

- 我能读懂设备是怎么通信的  
- 我能把底层命令抽象成更高层的动作模型  
- 我能把界面、线程、通信、执行调度组织起来  
- 我能在遇到问题的时候，不是停在那里，而是继续往下拆

到现在为止，这个上位机当然还没有结束。比如视觉标定我还没做完，AI Agent 也还需要继续优化，很多参数后面还值得继续外置化和模块化。但我已经很确定一件事：它不再只是一个“控制机械臂的小工具”了，而是一个真正被我一步步做起来的软件项目。
